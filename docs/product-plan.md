# 产品规划

## 1. 项目目标

本项目要解决的问题不是“做一个通用电商系统”，而是从一家真实线下小超市出发，完成从人工经营到数字化经营、再到线上化和智能化经营的改造。

阶段目标：

- 第一阶段：让店主能管理商品、查价格、记销售、看库存和临期提醒。
- 第二阶段：让店主能基于数据做补货、清仓、特价和经营复盘。
- 第三阶段：让顾客通过小程序查看特价、预留商品、到店自提。
- 第四阶段：让 AI 基于经营数据提供分析、建议和文案。

## 2. 核心用户

### 店主/经营者

主要诉求：

- 快速查商品价格和库存。
- 不再依赖记忆管理商品。
- 知道每天卖了多少钱、赚了多少钱。
- 及时发现缺货和临期商品。
- 能快速生成特价信息发到微信群/朋友圈。

### 家庭成员/店员

主要诉求：

- 操作简单，不需要懂复杂系统。
- 查价、录销售、看提醒要足够快。
- 页面适合电脑和手机使用。

### 顾客

后续小程序端用户。

主要诉求：

- 查看今日特价和到货信息。
- 预留商品，到店自提。
- 了解门店位置、营业时间和联系方式。

## 3. 业务痛点

### 商品价格难记

小超市商品数量多，且价格会随进货价和促销变化。店主和家人很难长期记住每个商品的售价、进价、库存和货架位置。

### 库存不清

很多门店只知道“货架上大概还有多少”，但不知道真实库存、库存成本和哪些商品快没了。

### 临期损耗

饮料、牛奶、零食、调料等商品都有保质期。没有系统提醒时，容易等到过期才发现。

### 缺少经营数据

店主可能知道今天收了多少钱，但不知道：

- 哪些商品最赚钱。
- 哪些商品卖得慢。
- 哪些商品占用了最多库存资金。
- 哪些品类应该补货。
- 哪些商品应该做特价处理。

### 线上化没有数据基础

如果没有内部商品库、库存和活动数据，直接做小程序会变成空壳商城。正确路线应先完成内部数字化，再把部分商品和活动开放给顾客端。

## 4. 产品边界

### 第一阶段做什么

- 商品管理
- 查价
- 销售记录
- 库存扣减
- 库存流水
- 低库存提醒
- 临期提醒
- 基础经营看板

### 第一阶段不做什么

- 不做完整 POS 收银系统。
- 不做复杂会员积分。
- 不做配送系统。
- 不做完整电商购物车。
- 不做多门店连锁能力。
- 不做财务报表和税务系统。

## 6. 核心业务流程

### 商品录入流程

```text
新增商品
→ 填写商品基本信息、价格、库存、到期日期
→ 保存商品
→ 写入初始库存流水
→ 商品可被查价、销售和统计
```

### 销售扣库存流程

```text
搜索/扫码商品
→ 加入销售单
→ 确认数量
→ 创建销售单和销售明细
→ 按销售数量扣减库存
→ 写入库存流水
→ 更新看板数据
```

### 临期处理流程

```text
商品录入时填写到期日期
→ 系统根据日期计算临期状态
→ 临期商品进入提醒列表
→ 店主设置特价或标记处理
→ 后续可同步到顾客端特价专区
```

### 小程序开放流程

```text
后台维护商品库
→ 店主选择部分商品加入特价活动
→ 小程序展示特价商品
→ 顾客提交预留
→ 后台处理预留
→ 顾客到店自提
```

### AI 经营分析流程

```text
店主输入经营问题
→ 系统识别需要的数据范围
→ 从数据库查询销售、库存、临期等数据
→ 大模型生成分析和建议
→ 保存 AI 建议
→ 店主采纳或忽略
```

## 7. 版本路线

### V0.1 商品与查价

- 商品 CRUD
- 商品搜索
- 快速查价
- 基础分类

验收标准：

- 能录入 30 个真实商品。
- 能在 3 秒内查到商品售价、库存和货架位置。

### V0.2 销售与库存

- 新增销售单
- 销售明细
- 自动扣库存
- 库存流水

验收标准：

- 完成一次销售后，库存自动减少。
- 库存流水能解释库存变化原因。

### V0.3 预警与看板

- 低库存提醒
- 临期提醒
- 今日销售额
- 今日毛利
- 库存成本

验收标准：

- 能看到哪些商品快没货。
- 能看到哪些商品快过期。
- 能看到当天销售和毛利。

### V0.4 活动与对外展示

- 特价活动
- 今日推荐
- 顾客端商品展示
- 到店自提预留

### V0.5 AI 经营助手

- 自然语言问数
- 补货建议
- 临期建议
- 活动文案生成
- AI 建议历史记录
